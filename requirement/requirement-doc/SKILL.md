# 需求规格说明书编写技能（SRD）

## 概述

需求规格说明书（Software Requirements Specification, SRS）是信息化项目成功的基石。不完整的需求文档会导致开发返工、进度延误、成本超支。

**核心原则**: 好的SRD应该是完整的、一致的、可验证的、可追踪的。坏的SRD会导致需求蔓延、理解偏差、交付失败。

## 何时使用

**始终：**
- 新项目启动时
- 系统重构前
- 功能重大变更时
- 项目验收标准不清晰时

**触发短语：**
- "写需求文档"
- "SRD怎么写"
- "需求规格说明"
- "功能需求"
- "非功能需求"
- "怎么才算需求完整"

## 需求文档核心结构

### 1. 引言部分
```
1.1 目的
    - 说明本文档的目的和范围
    
1.2 范围
    - 描述系统的主要功能
    - 明确边界和限制
    
1.3 定义和缩略语
    - 列出文档中使用的专业术语
```

### 2. 总体描述
```
2.1 产品背景
    - 项目背景和动机
    - 与其他系统的关系
    
2.2 用户特征
    - 最终用户画像
    - 用户权限分级
    
2.3 约束条件
    - 法规约束
    - 硬件约束
    - 技术约束
```

### 3. 功能需求

#### 3.1 功能需求模板
```
【FR-001】用户登录功能

描述：系统应允许已注册用户通过用户名和密码登录系统

前置条件：用户已在系统中注册

基本流程：
1. 用户输入用户名
2. 用户输入密码
3. 用户点击登录按钮
4. 系统验证凭证
5. 系统显示登录成功

备选流程：
- 3.1 若用户名不存在，显示"用户不存在"
- 3.2 若密码错误，显示"密码错误"，允许重试（最多3次）
- 3.3 若账户被锁定，显示"账户已锁定，请联系管理员"

后置条件：用户成功登录系统，跳转至首页

验收标准：
- [ ] 正确的用户名和密码可以登录成功
- [ ] 错误密码3次后账户被锁定
- [ ] 登录后Session有效期为2小时
```

#### 3.2 需求编号规范
```
FR-XXX  功能需求（Functional Requirement）
NFR-XXX 非功能需求（Non-Functional Requirement）
UR-XXX  用户需求（User Requirement）
BR-XXX  业务需求（Business Requirement）
```

### 4. 非功能需求

#### 4.1 性能需求
```
【NFR-001】响应时间
- 普通操作响应时间 ≤ 2秒
- 复杂查询响应时间 ≤ 5秒
- 文件上传响应时间 ≤ 10秒

【NFR-002】并发能力
- 支持500人同时在线
- 峰值并发数1000人
```

#### 4.2 安全需求
```
【NFR-010】认证授权
- 所有API需要认证访问
- 密码存储使用BCrypt加密
- Session超时时间≤120分钟

【NFR-011】数据安全
- 敏感数据加密存储
- 传输过程使用HTTPS
- 操作日志完整记录
```

#### 4.3 可用性需求
```
【NFR-020】易用性
- 关键操作≤3步完成
- 错误提示清晰易懂
- 支持键盘快捷操作
```

## 需求文档常见问题

### 问题1：需求描述模糊
```
❌ 错误示例：
"系统应该快速响应用户操作"

✅ 正确示例：
"系统普通操作响应时间应≤2秒，复杂查询响应时间应≤5秒（P99）"
```

### 问题2：缺少验收标准
```
❌ 错误示例：
"用户可以上传文件"

✅ 正确示例：
"用户可以上传文件
  - 支持格式：PDF、DOC、DOCX、XLS、XLSX
  - 文件大小限制：单个文件≤10MB
  - 同一批次最多上传10个文件
  - 上传进度实时显示
  - 上传成功后显示文件列表"
```

### 问题3：需求边界不清晰
```
❌ 错误示例：
"系统支持数据导出"

✅ 正确示例：
"系统支持导出以下数据：
  - 用户列表（支持CSV、Excel格式）
  - 业务单据（仅支持PDF格式）
  - 统计报表（支持PDF、Excel格式）
  
  导出限制：
  - 单次导出最多10000条记录
  - 超过时提示分批导出
  - 导出文件保留期限：7天"
```

## 需求文档质量检查清单

### 完整性检查
- [ ] 每个需求都有唯一编号
- [ ] 每个功能需求都有验收标准
- [ ] 涵盖了所有用户类型
- [ ] 包含正常流程和异常流程
- [ ] 明确标注优先级（P0/P1/P2）

### 一致性检查
- [ ] 术语定义一致
- [ ] 编号体系一致
- [ ] 格式风格一致
- [ ] 与上级需求一致

### 可验证性检查
- [ ] 需求可以被测试
- [ ] 验收标准明确量化
- [ ] 不存在二义性描述

## AI辅助需求编写技巧

### 提示词模板
```
请帮我编写以下功能的SRD：

功能名称：【功能名称】
用户角色：【目标用户】
业务背景：【业务场景】

请包含：
1. 功能描述
2. 用户流程（基本流程 + 备选流程）
3. 数据需求（输入/输出）
4. 验收标准（至少5条）
5. 异常处理

格式要求：使用【FR-XXX】编号，验收标准使用[ ]checkbox格式
```

### 需求评审提示词
```
请审查以下需求文档，检查以下问题：
1. 需求是否完整？
2. 是否有模糊描述？
3. 验收标准是否可测试？
4. 是否有遗漏的异常场景？
5. 优先级是否合理？

请逐条列出问题并给出修改建议。
```
